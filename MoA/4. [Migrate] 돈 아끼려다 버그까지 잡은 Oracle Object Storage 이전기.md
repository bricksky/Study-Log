> *"AWS 요금 고지서를 보고 나서야 무료가 얼마나 소중한지 깨달았다."*

프로젝트 초기, 퀴즈 이미지들을 별 고민 없이 AWS S3에 올렸다. 기능은 잘 돌아갔다. 문제는 돈이었다. Oracle Cloud에 서버를 올리면서 Object Storage 20GB를 **영구 무료**로 쓸 수 있다는 걸 뒤늦게 알았다. 어차피 같은 Oracle Cloud 인프라를 쓰는데, S3에 돈을 낼 이유가 없었다. 겸사겸사 코드를 들여다보니 공간력 퀴즈 셔플 로직에서 아찔한 버그도 발견했고, 코드 리뷰를 통해 미처 보지 못했던 문제들도 추가로 잡았다. 이번 글에서는 S3에서 Oracle Object Storage로 이전한 전 과정과, 정답이 엉뚱한 이미지를 가리키고 있던 버그 수정 과정을 정리한다.

**리팩토링 요약**

- **배경**: AWS S3 비용 발생 및 공간력 퀴즈 셔플 로직 버그 발견
- **핵심**: Oracle Object Storage로 이미지 이전, 셔플 후 **정답 번호 재계산** 및 Redis를 활용한 **채점 로직 개선**
- **결과**: 이미지 스토리지 비용 절감, 공간력 퀴즈 정답 처리 정상화, 예외 처리 및 쿼리 안정성 강화

### 1. S3 → Oracle Object Storage 마이그레이션

**전략: 서버에서 서버로 직접 옮기기**

이미지를 이전하는 방법은 여러 가지가 있지만, 가장 단순한 방법을 택했다. Oracle 서버에서 AWS CLI로 S3 이미지를 통째로 내려받고, OCI CLI로 Oracle Object Storage에 그대로 올리는 것이다. 로컬 맥을 거칠 필요도 없다.

```
S3 버킷(moa-bucket-s3) → Oracle 서버(임시 보관) → Oracle Object Storage(moa-bucket)
```

</br>

**STEP 1. AWS CLI 설치 및 S3 이미지 다운로드**

AWS CLI 설치부터 막혔다. `sudo apt-get install awscli`로 간단히 설치하려 했는데 패키지를 찾을 수 없다는 오류가 떴다. 이 부분은 AI의 도움을 받아 해결했는데, Oracle 서버가 ARM 기반 CPU를 쓰기 때문이었다. 일반적인 x86 리눅스용 패키지는 ARM 아키텍처에서 실행되지 않아서, AWS 공식 사이트에서 ARM용(`aarch64`) 패키지를 따로 받아야 했다. 쉽게 말해 같은 프로그램이라도 CPU 종류에 맞는 버전이 따로 존재한다는 것을 그때 처음 실감했다.

```bash
sudo apt-get install unzip -y
curl "https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip" -o "awscliv2.zip"
unzip awscliv2.zip && sudo ./aws/install
aws configure  # Access Key, Secret Key, region(ap-northeast-2) 입력
```

</br>

설정이 끝나면 S3 버킷을 통째로 서버로 내려받는다.

```bash
mkdir -p ~/moa-images
aws s3 sync s3://moa-bucket-s3/ ~/moa-images/
# 45개 이미지 다운로드 완료
```

</br>

**STEP 2. OCI CLI 설치 및 설정**

OCI CLI 설치 자체는 공식 스크립트 한 줄로 됐다. 문제는 그 다음이었다. `oci setup config`로 설정을 진행하다가 패스프레이즈 입력 단계에서 계속 튕기는 문제가 생겼다. Enter를 눌러도, `N/A`를 입력해도 같은 화면이 반복됐다. 결국 이 부분도 AI 도움을 받아 `oci setup config` 없이 키를 직접 생성하고 config 파일을 수동으로 작성하는 방식으로 우회했다.

OCI CLI는 Oracle Cloud에 접속할 때 **개인 키 파일**로 인증한다. 마치 SSH 접속할 때 `moa.key` 파일을 쓰는 것과 같다. 그런데 이 키 파일 자체가 탈취되면 누구든 Oracle Cloud에 접근할 수 있게 된다. 그래서 키 파일에 **추가로 비밀번호를 걸어두는 것**이 패스프레이즈다. 키 파일이 탈취되더라도 패스프레이즈를 모르면 사용할 수 없게 막아주는 역할을 한다.

```bash
bash -c "$(curl -L https://raw.githubusercontent.com/oracle/oci-cli/master/scripts/install/install.sh)"
source ~/.bashrc

openssl genrsa -out ~/.oci/oci_api_key.pem 2048
openssl rsa -pubout -in ~/.oci/oci_api_key.pem -out ~/.oci/oci_api_key_public.pem
openssl rsa -pubout -outform DER -in ~/.oci/oci_api_key.pem | openssl md5 -c
```

</br>

config 파일은 직접 작성했다. config 파일이란 OCI CLI가 Oracle Cloud에 접속할 때 필요한 인증 정보를 담아두는 설정 파일이다. 어떤 계정으로, 어떤 리전에, 어떤 키를 사용해서 접속할지를 여기에 명시해두면, 이후 `oci` 명령어를 실행할 때마다 이 파일을 참조해서 인증을 처리한다.

```
[DEFAULT]
user=ocid1.user.oc1...(생략)
fingerprint=b0:45:ea:22:55:bd:43:44:92:d4:a2:b8:07:50:65:ba
tenancy=ocid1.tenancy.oc1...(생략)
region=ap-chuncheon-1
key_file=/home/ubuntu/.oci/oci_api_key.pem
```

</br>

**STEP 3. Oracle Object Storage로 업로드**

```bash
for file in ~/moa-images/*; do
  filename=$(basename "$file")
  oci os object put --bucket-name moa-bucket --file "$file" --name "$filename" --force
done
# 45개 업로드 확인
oci os object list --bucket-name moa-bucket | grep '"name"' | wc -l
```
</br>

업로드 후 버킷 Visibility를 **Public**으로 변경했다. 안 하면 앱에서 이미지 URL로 접근할 때 403이 뜬다.

```bash
curl -I "https://objectstorage.ap-chuncheon-1.oraclecloud.com/n/axowe6gavgsk/b/moa-bucket/o/e9e32cdc-5_spacetime_1.png"
# HTTP/1.1 200 OK ✅
```

</br>

**STEP 4. DB URL 일괄 업데이트**

DB `quiz_question` 테이블의 `detail_data` 컬럼에 이미지 경로가 JSON 형태로 저장되어 있었다. JSON 안의 문자열을 바꿔야 했는데, 이 부분도 AI의 도움을 받았다. MySQL의 `REPLACE()` 함수는 컬럼 안에서 특정 문자열을 찾아 다른 문자열로 한 번에 치환해주는 함수다. 마치 텍스트 에디터의 "전체 바꾸기" 기능과 같다. JSON이 문자열로 저장된 컬럼이라도 `REPLACE()`로 URL 부분만 깔끔하게 바꿀 수 있다는 걸 이때 처음 알았다.

```sql
UPDATE quiz_question
SET detail_data = REPLACE(
  CAST(detail_data AS CHAR),
  's3://moa-bucket-s3/',
  'https://objectstorage.ap-chuncheon-1.oraclecloud.com/n/axowe6gavgsk/b/moa-bucket/o/'
)
WHERE detail_data LIKE '%s3://moa-bucket-s3/%';
-- 31 row(s) affected
```

</br>

### 2. 공간력 퀴즈 셔플 버그 수정

**문제 상황: "셔플을 했는데 왜 항상 같은 순서지?"**

공간력 퀴즈는 문제 이미지 1개와 보기 이미지 2개로 구성된다. DB에는 보기 배열(`imageOptionsUrl`)과 정답 번호(`answer: "1"` 또는 `"2"`)가 저장되어 있다. 매번 보기 순서를 랜덤하게 섞어 출제하는 게 의도였는데, 코드를 뜯어보니 버그가 2개나 있었다.

</br>

**버그 1: 셔플했는데 원본을 반환**

셔플된 배열(`shuffledOptions`)을 만들어놓고 정작 반환할 때는 셔플 안 된 원본(`processedOptions`)을 돌려주고 있었다. 셔플 코드 자체가 아무 의미 없이 실행만 되다가 버려지는 상태였다.

**[Before]**

```java
List<String> shuffledOptions = new ArrayList<>(processedOptions);
Collections.shuffle(shuffledOptions);  // 셔플은 했는데...

return new SpacetimeQuizQuestionDto(
        ...
        entity.getAnswer(),   // ❌ 원래 정답 번호 그대로
        processedQuestionImage,
        processedOptions      // ❌ 셔플 안 된 원본 반환
);
```

</br>

**버그 2: 셔플 후 정답 번호 재계산 안 함**

배열 순서가 바뀌면 정답 번호도 같이 바뀌어야 하는데, 이를 재계산하지 않았다.

|  | 배열 | 정답 번호 | 실제 정답 이미지 |
| --- | --- | --- | --- |
| 셔플 전 | `["_2.png", "_1.png"]` | `"2"` | _1.png ✅ |
| 셔플 후 (버그) | `["_1.png", "_2.png"]` | `"2"` | _2.png ❌ |
| 셔플 후 (수정) | `["_1.png", "_2.png"]` | `"1"` | _1.png ✅ |

**[After]**

```java
int originalAnswerIdx = Integer.parseInt(entity.getAnswer()) - 1;
String answerImageUrl = processedOptions.get(originalAnswerIdx);

List<String> shuffledOptions = new ArrayList<>(processedOptions);
Collections.shuffle(shuffledOptions);

int newAnswerIdx = shuffledOptions.indexOf(answerImageUrl) + 1;
String newAnswer = String.valueOf(newAnswerIdx);

return new SpacetimeQuizQuestionDto(
        ...
        newAnswer,        // ✅ 재계산된 정답 번호
        processedQuestionImage,
        shuffledOptions   // ✅ 셔플된 배열 반환
);
```

### 3. 코드 리뷰 후 추가 수정

PR을 올리고 나서 리뷰어로부터 3가지 지적을 받았다.

</br>

**문제 1: 채점 로직이 셔플을 모른다**

가장 치명적인 지적이었다. DTO에서 셔플 후 `newAnswer`를 반환하지만, 채점할 때 서버는 DB의 원본 `question.getAnswer()`와 사용자 제출값을 직접 비교하고 있었다.

예를 들어 사용자는 셔플된 화면 기준으로 `"1"`을 제출했는데, 서버는 DB의 원본 `"2"`와 비교해서 오답 처리하는 상황이었다.

해결 방법을 고민했다. 셔플은 요청할 때마다 랜덤으로 달라지기 때문에 채점 시 서버가 어떤 순서로 셔플됐는지 알 수 없다. 결국 퀴즈 출제 시 셔플된 정답을 Redis에 저장해두고, 채점 시 꺼내서 비교하는 방식으로 해결했다.

</br>

**QuizQuestionServiceImpl.java** - 출제 시 Redis 저장:

```java
private static final String SPACETIME_ANSWER_KEY = "spacetime:answer:%d:%d";
private static final long SPACETIME_ANSWER_TTL = 60 * 24; // 24시간

// 퀴즈 출제 시
QuizQuestionDto dto = quizConverter.toDto(e);
if (type == EQuizType.SPACETIME && dto instanceof SpacetimeQuizQuestionDto spacetimeDto) {
    String key = String.format(SPACETIME_ANSWER_KEY, userId, spacetimeDto.questionId());
    redisService.setData(key, spacetimeDto.answer(), SPACETIME_ANSWER_TTL);
}
```

</br>

**QuizResultServiceImpl.java** - 채점 시 Redis에서 조회:

```java
private boolean isAnswerCorrect(QuizQuestion question, String userAnswer, Long userId) {
    if (question.getType() == EQuizType.SPACETIME) {
        String key = String.format(SPACETIME_ANSWER_KEY, userId, question.getId());
        String shuffledAnswer = redisService.getData(key, String.class);
        if (shuffledAnswer != null) {
            return shuffledAnswer.trim().equalsIgnoreCase(userAnswer.trim());
        }
        // Redis에 없으면 원본 answer로 채점 (fallback)
    }
    return question.getAnswer().trim().equalsIgnoreCase(userAnswer.trim());
}
```

</br>

**문제 2: 예외 처리 누락으로 500 에러**

`JsonProcessingException`만 잡고 있었는데, 아래 두 곳에서 다른 예외가 날 수 있었다.

```java
Integer.parseInt(entity.getAnswer())    // NumberFormatException 가능
processedOptions.get(originalAnswerIdx) // IndexOutOfBoundsException 가능
```

catch 블록을 추가했다.

```java
} catch (JsonProcessingException e) {
    throw new CustomException(QuizErrorCode.QUIZ_DATA_FORMAT_ERROR);
} catch (NumberFormatException | IndexOutOfBoundsException e) {
    throw new CustomException(QuizErrorCode.QUIZ_DATA_FORMAT_ERROR);
}
```

</br>

**문제 3: 네이티브 쿼리 컬럼명 불일치**

`QuizQuestion` 엔티티에서 물리 컬럼명을 `quiz_question_id`, `quiz_type`으로 바꿨는데, `QuizQuestionRepository`의 네이티브 쿼리는 아직 `qq.id`, `qq.type`을 쓰고 있었다. 이 상태면 오늘의 퀴즈 ID 조회와 타입별 퀴즈 조회가 런타임에 깨진다.

**[Before]**

```sql
SELECT qq.id FROM quiz_question qq WHERE qq.type = :type
```

**[After]**

```sql
SELECT qq.quiz_question_id FROM quiz_question qq WHERE qq.quiz_type = :type
```

### 4. 배포 후 발견된 추가 이슈: 문제와 보기 이미지 불일치

배포 후 앱에서 공간력 퀴즈를 확인해보니 문제 이미지와 보기 이미지가 전혀 엉뚱한 조합으로 표시되고 있었다. 예를 들어 1번 문제의 이미지가 나오는데, 보기에는 2번, 3번 문제의 이미지가 붙어있는 식이었다.

혹시 DB 데이터가 잘못된 건 아닌지 직접 확인해봤다.

```sql
SELECT
    id,
    answer,
    JSON_UNQUOTE(JSON_EXTRACT(detail_data, '$.questionImageUrl')) as question_image,
    JSON_UNQUOTE(JSON_EXTRACT(detail_data, '$.imageOptionsUrl[0]')) as option_1,
    JSON_UNQUOTE(JSON_EXTRACT(detail_data, '$.imageOptionsUrl[1]')) as option_2
FROM quiz_question
WHERE type = 'SPACETIME'
ORDER BY id;
```

결과를 보면 파일명 패턴으로 확인할 수 있었다. 예를 들어 id=130의 경우 `questionImageUrl`은 `spacetime_1.png`, 보기 이미지들은 `spacetime_1_1.png`, `spacetime_1_2.png`로 같은 세트 번호(`_1`)를 공유하고 있었다. DB 데이터 자체는 올바르게 묶여있었다.

백엔드 API 응답도 `200 OK`로 정상이었고, 로그에서도 각 문제 객체 안에 `questionImageUrl`과 `imageOptionsUrl`이 함께 담겨 반환되고 있었다.

결론은 **클라이언트 문제**였다. 백엔드는 각 문제마다 아래 구조로 데이터를 묶어서 내려주고 있는데, 프론트엔드에서 문제 이미지 배열과 보기 이미지 배열을 분리해서 따로 인덱싱하는 방식으로 렌더링하고 있었던 것이다.

```json
{
  "questionId": 130,
  "quizType": "SPACETIME",
  "questionImageUrl": "...(문제 이미지)",
  "imageOptionsUrl": ["...(보기1)", "...(보기2)"],
  "answer": "2"
}
```

백엔드 기준에서는 데이터가 문제없이 묶여있었지만, 흐름 전체를 끝까지 추적하지 않으면 이런 부분에서 구멍이 생긴다는 걸 다시 한번 느꼈다.

### 5. 결론

이번 작업을 하면서 솔직히 AI 없이는 못 했을 부분들이 있었다. ARM 아키텍처 이슈, OCI CLI 패스프레이즈 우회, MySQL JSON 컬럼 치환 방법 모두 혼자였다면 한참 헤맸을 것들이다. 아직 이 개념들을 완전히 내 것으로 소화했다고는 말 못 하지만, 적어도 어떤 문제가 있었고 어떻게 해결됐는지는 이해했다.

셔플 버그는 달랐다. 코드를 직접 읽으면서 스스로 의문을 품었고, 수정 방향도 직접 잡았다. 그런데 코드 리뷰에서 내가 놓친 것들이 또 나왔다. DTO에서 정답을 바꿔도 채점 로직은 여전히 원본을 보고 있었다는 사실을 리뷰 전까지 몰랐다. 배포 후에도 클라이언트 렌더링 문제가 뒤늦게 발견됐다. 기능이 돌아가 보여도 흐름 전체를 따라가지 않으면 구멍이 생긴다는 걸 이번에 다시 배웠다.
