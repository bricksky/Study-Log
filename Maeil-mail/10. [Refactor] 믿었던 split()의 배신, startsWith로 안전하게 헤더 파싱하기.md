### **공백 하나의 차이 — `split(" ")` 에서 `startsWith` + `substring` 으로**

> 🔗 **관련 PR**: [어드민 인증 헤더 파싱 시 발생하는 배열 인덱스 취약점을 개선한다. ](https://github.com/maeil-mail/maeil-mail-be/pull/306)

 어드민 API 인증 헤더 검증 방식을 `split(" ")` 기반 배열 분리에서, `startsWith("Basic ")`와 `substring`을 활용한 구조로 전환했다. 기존 방식은 클라이언트가 공백 없이 헤더를 전송할 경우 `ArrayIndexOutOfBoundsException`이 발생하는 취약점을 가지고 있었다. 이를 개선하는 과정에서 불필요한 검증 메서드 두 개도 함께 제거되어 코드가 훨씬 단순해졌다.



---

**리팩토링 요약**

- **배경:** 공백 없는 헤더 입력 시 배열 접근에서 런타임 예외 발생 가능성 존재
- **핵심:** `startsWith("Basic ")`으로 공백까지 포함한 형식 검증, `substring`으로 키값 추출
- **결과:** `validateAuthType`, `validateIdentity` 메서드 제거 및 예외 방어 강화



</br>


### **1. 배경 지식: HTTP Authorization 헤더 구조**

어드민 API는 요청마다 아래 형식의 HTTP 헤더를 통해 인증을 수행한다.

> Authorization: Basic my-secret-key
> 
- `Authorization` — 헤더 이름
- `Basic` — 인증 타입을 나타내는 **고정값**. HTTP 표준 스펙에서 클라이언트가 Basic 인증 방식으로 요청할 때 반드시 붙여야 한다.
- `my-secret-key` — 실제 인증 키값. 이 프로젝트에서는 `@Value("${admin.secret}")`으로 주입된 `adminSecret`과 일치해야 어드민 접근이 허용된다.

즉 헤더는 `Basic`(고정) + 공백 + 키값의 구조로 이루어져 있으며, 공백이 두 값을 구분하는 기준점이 된다.

</br>

### **2. 문제 상황: 공백에 의존하는 파싱의 불안정성**

기존 코드는 공백을 기준으로 `split(" ")`하여 배열로 분리한 뒤, 0번 인덱스에서 인증 타입을, 1번 인덱스에서 키값을 꺼내 각각 검증하는 방식을 사용했다.

**<기존 코드>**

```java
`String[] authTokens = header.split(" ");
String authType = authTokens[0];  // "Basic"
String secret = authTokens[1];    // 실제 키값

validateAuthType(authType);
validateIdentity(secret);

private void validateAuthType(String authType) {
    if (!("Basic".equals(authType))) {
        throw new IllegalArgumentException("지원하지 않는 인증 타입입니다.");
    }
}

private void validateIdentity(String secret) {
    if (!(adminSecret.equals(secret))) {
        throw new IllegalArgumentException(ADMIN_AUTH_FAIL_MESSAGE);
    }
}`
```

이 방식은 `Authorization: Basicmy-secret-key` 처럼 공백 없이 헤더가 들어올 경우, `split(" ")`의 결과가 길이 1짜리 배열이 되어 `authTokens[1]` 접근 시 `ArrayIndexOutOfBoundsException`이 발생한다. 클라이언트의 실수 하나가 서버 예외로 이어질 수 있는 구조다.

</br>

### **3. 해결: 형식 자체를 검증하고 값을 추출**

`"Basic "` (공백 포함)이라는 접두사 전체를 검증 기준으로 삼으면, 공백 누락 여부까지 한 번에 걸러낼 수 있다. 이후 `substring`으로 해당 접두사의 길이만큼 잘라내면 키값만 정확히 추출된다.

- `substring(int beginIndex)` — 문자열에서 지정한 인덱스부터 끝까지 잘라내는 메서드.
- `"Basic ".length()`는 6이므로 `substring(6)`은 6번 인덱스 이후 문자열, 즉 키값만 반환한다.

> "Basic my-secret-key"
> 
> 
> 0123456...
> 
> substring(6) → "my-secret-key"
> 

**<수정 코드>**

```java
`if (header == null || !header.startsWith("Basic ")) {
    throw new IllegalArgumentException(ADMIN_AUTH_FAIL_MESSAGE);
}

String secret = header.substring("Basic ".length());
if (!adminSecret.equals(secret)) {
    throw new IllegalArgumentException(ADMIN_AUTH_FAIL_MESSAGE);
}`
```

- `startsWith("Basic ")` — `"Basic "` (공백 포함 6글자) 전체를 검증하므로 공백 누락도 이 단계에서 차단된다.
- `substring("Basic ".length())` — 6번 인덱스 이후 문자열만 추출하므로 키값만 정확히 꺼낸다.
- `validateAuthType`, `validateIdentity` 두 메서드는 로직이 인라인으로 흡수되어 삭제됐다.

</br>

### **4. 실수 포인트: "Basic"과 "Basic " 는 다르다**

수정 과정에서 `startsWith`와 `substring` 모두에 `"Basic"`(공백 없음)을 사용하는 실수를 했다.

| **위치** | **잘못된 코드** | **문제** |
| --- | --- | --- |
| `startsWith` | `startsWith("Basic")` | 공백 없는 `Basicmy-secret-key`도 통과됨 |
| `substring` | `substring("Basic".length())` | 5글자만 잘라내므로 `" my-secret-key"`(공백 포함)가 남아 `adminSecret`과 불일치하여 인증 항상 실패 |

두 곳 모두 반드시 `"Basic "` (공백 포함)을 사용해야 한다. `startsWith`에서는 형식 보장을, `substring`에서는 정확한 키값 추출을 위해서다.

</br>

### **5. 왜 이렇게 설계했는가?**

**5.1 배열 접근 제거로 런타임 예외 원천 차단**

`split` 후 인덱스로 접근하는 방식은 입력 형식이 조금이라도 어긋나면 예외가 발생한다. `startsWith` + `substring` 방식은 형식 검증과 값 추출이 순차적으로 이루어지므로, 잘못된 입력은 검증 단계에서 명시적인 `IllegalArgumentException`으로 처리된다.

**5.2 검증 책임의 통합**

기존에는 `validateAuthType`과 `validateIdentity` 두 메서드에 검증 책임이 분산되어 있었다. 변경 후에는 null 체크, 형식 검증, 키값 일치 검증이 `preHandle` 안에서 순서대로 읽히므로 코드의 흐름이 선형적으로 파악된다.

</br>

### **6. 결론: 방어적 파싱이 안정적인 인증을 만든다**

단순해 보이는 문자열 파싱도 입력 형식에 대한 방어 없이는 언제든 예외로 이어질 수 있다. 이번 리팩토링은 공백 하나의 누락이라는 사소한 상황을 계기로, 입력을 신뢰하지 않고 명시적으로 검증하는 방어적 설계의 중요성을 체감한 사례다. 또한 수정 과정에서 `"Basic"`과 `"Basic "` 한 글자 차이가 인증 로직 전체를 무너뜨릴 수 있다는 점도 직접 경험했다.

</br>
